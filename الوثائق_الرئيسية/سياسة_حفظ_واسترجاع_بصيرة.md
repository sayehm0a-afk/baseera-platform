# سياسة حفظ واسترجاع منصة بصيرة

**النسخة:** 1.0
**التاريخ:** 14 يوليو 2026
**المؤلف:** Manus AI (بصفتي كبير مهندسي البرمجيات)

## 1. المقدمة

تحدد هذه الوثيقة سياسة حفظ واسترجاع مشروع منصة "بصيرة"، وتهدف إلى ضمان عدم فقدان المشروع تحت أي ظرف من الظروف، وتوفير إرشادات واضحة لاستعادة المشروع بالكامل من نسخ احتياطية خارجية ومستقلة. هذه السياسة جزء لا يتجزأ من استراتيجية استمرارية الأعمال للمنصة.

## 2. مبادئ أساسية

*   **عدم فقدان البيانات:** يجب أن تكون جميع الأصول الهامة للمشروع (الكود، الوثائق، القرارات) محمية ضد الفقدان.
*   **قابلية الاسترجاع:** يجب أن يكون المشروع قابلاً للاسترجاع بالكامل وبسهولة من أي نقطة حفظ معتمدة.
*   **الاستقلالية:** يجب أن تكون النسخ الاحتياطية مستقلة عن بيئة التطوير الأصلية.
*   **الأمان:** يجب حماية النسخ الاحتياطية ومستودعات الكود من الوصول غير المصرح به.
*   **التوثيق:** يجب توثيق جميع إجراءات الحفظ والاسترجاع بشكل واضح.

## 3. استراتيجية الحفظ الرئيسية

### 3.1 المستودع الخارجي الرئيسي (GitHub)

*   **المنصة:** GitHub هو المستودع الخارجي الرئيسي والوحيد المعتمد لحفظ الكود المصدري، الوثائق، والتكوينات.
*   **النوع:** يجب أن يكون المستودع **خاصاً (Private)** لضمان أمان وحماية ملكية المشروع.
*   **اسم المستودع:** `baseera-platform`.
*   **الفرع الرئيسي:** `main`.

### 3.2 نقاط الحفظ الرسمية (Commits)

*   **التردد:** يجب إنشاء Commit بعد كل خطوة تطوير معتمدة أو تغيير ذي معنى في الكود أو الوثائق.
*   **رسالة Commit:** يجب أن تكون رسالة Commit واضحة، وصفية، وتتبع اتفاقيات التسمية المحددة في وثيقة Platform Engineering Blueprint.

### 3.3 علامات الإصدار (Tags)

*   **التردد:** يجب إنشاء Tag ثابت (Immutable Tag) بعد كل مرحلة رئيسية من مراحل المشروع أو عند إصدار نسخة جديدة.
*   **اتفاقية التسمية:** يجب أن تتبع Tags نظام الإصدار الدلالي (Semantic Versioning) أو اتفاقية تسمية المراحل (مثال: `v0.1.0-foundation`).
*   **الغرض:** تسهيل الرجوع إلى حالات مستقرة ومحددة من المشروع.

### 3.4 النسخ الاحتياطية الكاملة (ZIP Archives)

*   **التردد:** يجب إنشاء نسخة ZIP كاملة للمشروع عند كل إصدار مهم أو عند اكتمال مرحلة تأسيسية كبرى.
*   **المحتوى:** يجب أن تحتوي نسخة ZIP على جميع ملفات المشروع باستثناء الملفات المؤقتة أو الأسرار.
*   **التسمية:** يجب أن تكون تسمية ملف ZIP واضحة وتعكس الإصدار أو المرحلة (مثال: `baseera-foundation-v0.1.0.zip`).
*   **التحقق:** يجب إنشاء SHA-256 checksum لملف ZIP لضمان سلامة الملف عند الاسترجاع.
*   **ملف Manifest:** يجب إنشاء ملف Manifest يسرد جميع الملفات الموجودة داخل نسخة ZIP لسهولة التحقق من المحتوى.

### 3.5 الحفظ السحابي المستقل (Cloud Storage)

*   **المنصات:** يجب حفظ نسخة ZIP الاحتياطية على خدمة تخزين سحابي مستقلة (مثل iCloud Drive أو Google Drive) يمتلكها المستخدم.
*   **الاستقلالية:** يضمن هذا الحفظ السحابي استقلالية النسخة الاحتياطية عن بيئة التطوير وعن GitHub نفسه.
*   **الأمان:** يجب أن تكون هذه الخدمات مؤمنة بكلمات مرور قوية ومصادقة متعددة العوامل (MFA).

## 4. إدارة الأسرار

*   **عدم الحفظ في Git:** يجب عدم حفظ أي أسرار (مثل مفاتيح API، كلمات المرور، الشهادات) داخل مستودع Git أو في ملفات النسخ الاحتياطي ZIP.
*   **إدارة الأسرار المخصصة:** يجب استخدام خدمات إدارة الأسرار المخصصة (مثل AWS Secrets Manager, Azure Key Vault, HashiCorp Vault) في بيئات الإنتاج.
*   **ملفات .env:** في بيئات التطوير، يمكن استخدام ملفات `.env` المحلية، مع التأكد من إضافتها إلى `.gitignore`.

## 5. اختبار الاسترجاع الدوري

*   **التردد:** يجب إجراء اختبارات استرجاع دورية (على الأقل مرة كل ربع سنة) للتأكد من أن عملية الاسترجاع تعمل بشكل صحيح وأن النسخ الاحتياطية صالحة.
*   **السيناريو:** يجب أن يحاكي الاختبار سيناريو استرجاع المشروع بالكامل على جهاز جديد أو بيئة نظيفة.
*   **التحقق:** يجب التحقق من وجود جميع الملفات، الوثائق، والـ Tags بعد الاسترجاع.

## 6. إجراءات الاسترجاع

### 6.1 استرجاع المشروع من جهاز جديد (باستخدام GitHub)

1.  **تثبيت Git و GitHub CLI:** تأكد من تثبيت Git و GitHub CLI على الجهاز الجديد.
2.  **المصادقة مع GitHub:** قم بتسجيل الدخول إلى GitHub CLI باستخدام حسابك.
3.  **استنساخ المستودع:** `gh repo clone sayehm0a-afk/baseera-platform` (أو اسم المستخدم الخاص بك).
4.  **التحقق:** `git tag -l` للتحقق من وجود الـ Tags، و `ls -R` للتحقق من الملفات.

### 6.2 الرجوع إلى إصدار سابق (باستخدام Git)

1.  **عرض الـ Tags:** `git tag -l` لعرض جميع الـ Tags المتاحة.
2.  **التحقق من سجل Commits:** `git log --oneline` لتحديد Commit معين.
3.  **الرجوع إلى Tag:** `git checkout v0.1.0-foundation` (لاسترجاع حالة المشروع عند هذا الـ Tag).
4.  **الرجوع إلى Commit:** `git checkout <commit_hash>` (لاسترجاع حالة المشروع عند Commit معين).
