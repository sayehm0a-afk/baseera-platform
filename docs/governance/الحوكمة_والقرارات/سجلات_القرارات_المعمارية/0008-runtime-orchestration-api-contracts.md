# ADR 0008: Runtime & Orchestration API Contracts Design

## 1. العنوان

تصميم عقود واجهة برمجة التطبيقات (API Contracts) الموحدة للتفاعل مع Runtime & Orchestration.

## 2. تاريخ القرار

14 يوليو 2026

## 3. الحالة

مقبول (Accepted)

## 4. السياق

تتطلب منصة "بصيرة" واجهات برمجة تطبيقات (APIs) واضحة وموثقة جيدًا للسماح للمكونات الخارجية، أو حتى المكونات الداخلية الأخرى، بالتفاعل مع محرك التشغيل والتنسيق. يجب أن تضمن هذه الواجهات الاتساق، قابلية الاكتشاف، وسهولة الاستخدام، مع توفير آليات للتحقق من الصحة وإدارة الأخطاء. تحليل الفجوة (Runtime_Orchestration_Gap_Analysis_v1.0) أظهر أن عقود API الموحدة كانت مفقودة.

## 5. القرار

سيتم تصميم وتنفيذ عقود واجهة برمجة تطبيقات موحدة (API Contracts) باستخدام مواصفات OpenAPI (Swagger) لتوثيق جميع نقاط النهاية (endpoints). ستضمن هذه العقود:

-   **نقاط نهاية RESTful:** تصميم واجهات برمجة تطبيقات تتبع مبادئ REST لسهولة الاستخدام والتكامل.
-   **توثيق شامل:** استخدام OpenAPI Specification لتوفير وصف آلي وبشري لجميع واجهات برمجة التطبيقات، بما في ذلك المدخلات، المخرجات، رموز الحالة، والأخطاء.
-   **التحقق من صحة المدخلات والمخرجات:** تطبيق آليات قوية للتحقق من صحة البيانات لضمان سلامة النظام.
-   **إدارة الأخطاء الموحدة:** توفير استجابات أخطاء متسقة وواضحة للمطورين.
-   **إدارة الإصدارات (Versioning):** دعم إدارة إصدارات API لضمان التوافق مع الإصدارات السابقة (Backward Compatibility) عند إجراء تغييرات.

## 6. الأسباب

-   **سهولة التكامل (Ease of Integration):** تسهل على المطورين الخارجيين والداخليين التفاعل مع المنصة.
-   **الاتساق (Consistency):** تضمن أن جميع واجهات برمجة التطبيقات تتبع نفس المعايير والمبادئ.
-   **قابلية الاكتشاف (Discoverability):** توفر توثيقًا شاملاً يسهل فهم واستخدام واجهات برمجة التطبيقات.
-   **الموثوقية (Reliability):** تضمن سلامة البيانات من خلال التحقق من الصحة وإدارة الأخطاء.
-   **قابلية التوسع (Scalability):** تسمح بتوسيع الواجهات بشكل مستقل عن منطق العمل الأساسي.

## 7. البدائل التي تم النظر فيها

-   **GraphQL:** تم رفضه في هذه المرحلة بسبب تعقيده الأولي مقارنة بـ REST، مع إمكانية النظر فيه في المستقبل إذا تطلبت متطلبات الاستعلام المرن ذلك. REST يوفر توازناً جيداً بين البساطة والقوة في هذه المرحلة.
-   **gRPC:** تم رفضه بسبب الحاجة إلى تعريفات Protobuf، مما يزيد من التعقيد في بيئة Python، مع إمكانية النظر فيه للاتصالات الداخلية عالية الأداء في المستقبل.

## 8. التبعيات

-   تتطلب تعريفًا واضحًا لنماذج البيانات (Data Models) التي سيتم استخدامها في الطلبات والاستجابات.
-   تتطلب دمج مكتبة أو إطار عمل يدعم OpenAPI (مثل FastAPI مع Pydantic).

## 9. الآثار المترتبة

-   يتطلب وقتًا إضافيًا لتصميم وتوثيق واجهات برمجة التطبيقات.
-   يجب صيانة التوثيق باستمرار ليعكس أحدث التغييرات.
