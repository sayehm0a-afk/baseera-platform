# خطة نشر Basirah في الإنتاج

**التاريخ:** 18 يوليو 2026
**المؤلف:** Manus AI

---

## ملخص

تحدد هذه الوثيقة خطة النشر التفصيلية لمنصة Basirah في بيئة إنتاج حقيقية. تغطي الخطة متطلبات الخادم، وتكوينات البرامج، واعتبارات الأمان، والاستراتيجيات التشغيلية لضمان نشر ناجح ومستقر.

---

## 3. الأمان (Security Considerations)

-   **SSL/TLS:** يجب تكوين شهادات SSL/TLS لجميع الاتصالات الخارجية (مثل واجهة برمجة تطبيقات FastAPI) لضمان التشفير. يوصى باستخدام Let's Encrypt أو شهادات من مرجع مصدق موثوق به.
-   **جدار الحماية (Firewall):** تكوين جدار حماية صارم للسماح فقط بالوصول الضروري. يجب أن يكون منفذ التطبيق (8000) متاحًا للوصول الخارجي (عبر Load Balancer/Ingress)، بينما يجب أن تكون منافذ PostgreSQL (5432) و Redis (6379) متاحة فقط داخليًا للتطبيق.
-   **إدارة الأسرار:** استخدام حل إدارة الأسرار المخصص (مثل Kubernetes Secrets، HashiCorp Vault، AWS Secrets Manager) لتخزين `OPENAI_API_KEY` و `DATABASE_URL` و `REDIS_PASSWORD` وغيرها من البيانات الحساسة. يجب عدم تخزين الأسرار في الكود أو ملفات التكوين المكشوفة.
-   **تحديثات الأمان:** تطبيق تحديثات الأمان بانتظام لنظام التشغيل وجميع البرامج المثبتة (Docker، PostgreSQL، Redis، Python Packages).
-   **الوصول بأقل امتياز:** تكوين المستخدمين والخدمات بأقل الامتيازات المطلوبة لأداء وظائفهم.

## 4. النسخ الاحتياطي والاستعادة (Backup and Recovery)

-   **PostgreSQL:** تنفيذ استراتيجية نسخ احتياطي منتظمة لقاعدة بيانات PostgreSQL (مثل النسخ الاحتياطي الكامل اليومي، النسخ الاحتياطي التزايدي، سجلات WAL). يجب تخزين النسخ الاحتياطية في موقع آمن وخارج الموقع.
-   **Redis:** تكوين مثابرة Redis (RDB أو AOF) لضمان استمرارية البيانات. يجب أيضًا تنفيذ نسخ احتياطي منتظم لبيانات Redis.
-   **بيانات التطبيق:** نسخ احتياطي لأي بيانات تطبيق مهمة أو ملفات تكوين مخصصة.
-   **خطة التعافي من الكوارث (DRP):** تطوير واختبار خطة شاملة للتعافي من الكوارث لضمان استعادة سريعة وفعالة للنظام في حالة الفشل الكارثي.

## 5. المراقبة والتسجيل (Monitoring and Logging)

-   **مراقبة التطبيق:** استخدام أدوات مراقبة الأداء (APM) لمراقبة صحة التطبيق، ومقاييس الأداء (CPU، الذاكرة، زمن الاستجابة)، ومعدلات الأخطاء. يوصى بـ Prometheus و Grafana أو حلول سحابية مثل AWS CloudWatch.
-   **مراقبة البنية التحتية:** مراقبة صحة وأداء الخوادم وقاعدة البيانات و Redis.
-   **التسجيل المركزي:** تجميع سجلات التطبيق والبنية التحتية في نظام تسجيل مركزي (مثل ELK Stack - Elasticsearch, Logstash, Kibana - أو Loki/Grafana). يجب تكوين التنبيهات للأحداث الحرجة.
-   **فحوصات الصحة:** التأكد من أن نقاط نهاية فحص الصحة (`/health`) تعمل بشكل صحيح وتوفر معلومات دقيقة عن حالة المكونات.

## 6. قابلية التوسع (Scaling)

-   **التطبيق:** يمكن توسيع نطاق تطبيق FastAPI أفقيًا عن طريق زيادة عدد النسخ المتماثلة (replicas) في Kubernetes أو Docker Compose. يجب أن يكون التطبيق عديم الحالة (stateless) قدر الإمكان.
-   **PostgreSQL:** يمكن توسيع نطاق PostgreSQL عموديًا (بزيادة موارد الخادم) أو أفقيًا (باستخدام النسخ المتماثلة للقراءة أو حلول التجزئة) حسب الحاجة.
-   **Redis:** يمكن توسيع نطاق Redis عموديًا أو أفقيًا (باستخدام Redis Cluster) حسب حجم البيانات ومعدل النقل.

## 7. النشر (Deployment)

-   **CI/CD Pipeline:** تنفيذ خط أنابيب تكامل مستمر/نشر مستمر (CI/CD) لأتمتة بناء صور Docker واختبارها ونشرها في بيئات الإنتاج.
-   **Kubernetes:** استخدام Kubernetes manifests للنشر وإدارة دورة حياة التطبيق في بيئات الإنتاج.
-   **Docker Compose:** يمكن استخدام Docker Compose للبيئات الأصغر أو بيئات التطوير/التجريب.
